# 给 Codex（Plan Mode）的项目提示词 —— 情侣空间

> 使用方式：把【提示词正文】整段复制进 Codex 的 Plan 模式。先让它出计划，你确认/调整后再分阶段编码。

---

## 提示词正文

我要做一个面向情侣双人使用的私密空间应用，命名暂定"我们的空间"（可由你建议更好的名字）。核心目标：不管异地还是同城，两个人都能随时感受到对方的存在感和爱意，并且有持续不断的互动话题。产品调性要求"高级感"——温馨有爱但不能廉价、卡通化、滥用粉色爱心堆砌，要像一个精致的私人产品，而不是普通社交App。

请先进入 Plan 模式，不要直接写代码。先输出：总体架构、目录结构、技术选型对比及理由、分阶段任务拆解（含每阶段验收标准）、风险点清单。我确认后再开始编码。

### 一、产品形态与基本机制

- 这是**两人专属**的私密空间：通过邀请码/二维码完成一次性"配对"绑定，配对后系统里所有内容默认只对这两人可见，没有"广场""推荐"之类的公共流量入口。
- 需要考虑后续如果开源或者上线给其他情侣使用，是**多租户**的：每一对情侣的数据要严格隔离，绝不能有任何接口/页面能让 A 情侣看到 B 情侣的数据，包括管理后台也不能随意明文浏览用户私密内容（这点请在权限设计里明确写出来）。
- 优先做成**移动端为主**的体验（人在外面随手记录"在干什么"），技术选型上请对比：① uniapp/Vue3 跨端（一套代码出 H5 + 小程序 + App）；② 纯微信小程序（国内情侣类产品天然适合小程序，传播成本低，但功能和相机/相册权限上有限制）；③ React Native/Flutter 原生体验更好但开发成本更高。请给出你的推荐和理由，我倾向于优先方案①或②，因为我熟悉 Vue3。

### 二、核心功能（按优先级分层）

**P0（MVP）**
1. 账号体系 + 配对机制（邀请码绑定，解绑/换绑流程也要设计）
2. **相册**：双方都可上传照片/小视频，按时间线呈现，支持给照片加文字小注解（类似"这张是我们第一次旅行"）
3. **状态/动态**：随手发一条"我在干什么"，支持纯文字、配图、心情标签（比如"在加班""在想你""刚撸完猫"），对方收到推送
4. **日记**：可写私密日记（仅自己可见）或共享日记（双方可见可互相留言），支持基础排版（图文混排）
5. **纪念日与倒计时**：在一起天数自动计算并放在首页显眼位置；支持双方各自添加多个纪念日（恋爱纪念日、第一次见面、生日等），每个纪念日可设提醒时间和小卡片样式，临近时首页展示倒计时
6. 响应式/移动端优先的视觉风格，先定一套设计语言（见下面"设计与氛围"要求）

**P1（增强互动）**
1. **状态分享**：类似微信"状态"的机制——提供一组预设的生活状态（比如"在上班""在摸鱼""在想你""刚吃饭""睡了"），用户从里面选一个，可选配一张照片，一键发布给对方看。这是**完全自愿、随时可改、可不发**的轻量动作，不设任何"对方多久没更新状态就提醒/催促"的机制，不做"在线/最后活跃时间"这类会带来压力感的展示，核心体验是"我想分享的时候点一下就好"，而不是任务或义务。对方看到新状态后可以一键回应一个小动效（比如"抱抱""收到啦"），但回应也完全自愿，不强制、不提醒未回应。
2. **订单/心意分享**：记录"我给你点了什么外卖""买了什么小礼物"，做成手动登记的卡片（标题、金额可选、配图、留言），**不要设计成自动同步美团/饿了么账号订单**——这类平台不对第三方开放个人订单读取接口，技术上做不到也涉及账号安全风险；正确做法是让用户截图/手动填写后生成一张好看的"心意卡片"
3. **共同待办/愿望清单**：双方共用一个列表，随手加"想一起做的事/要一起买的东西"（比如"一起去看海""周末囤的纪录片""攒钱买的相机"），任意一方可勾选完成。**不做任务指派/截止日期提醒这类管理工具式的功能**，保留"一起的心愿单"的轻松感而不是待办事项的紧迫感；完成时给一个小庆祝动效，可选支持把完成的心愿一键转存成一条回忆相册记录
4. 互动激励机制：每日一句"今日话题"卡片（系统随机推送一个轻松的提问，比如"如果现在能瞬间到对方身边，你想做什么"），双方都答了才解锁查看对方的答案，制造"对话钩子"
5. 时间胶囊/未来信：写一封定时发送给对方的信（比如设定纪念日才能打开）
6. "回忆推送"：类似"一年前的今天"，自动把当天历史的照片/日记推给双方
7. 消息推送（站内 + 微信服务通知/小程序订阅消息，看选型）

**P2（开源/上线准备）**
1. 隐私与数据保护作为独立任务列出：
   - 日记等敏感字段加密存储（至少做到数据库层加密或应用层加密）
   - 照片存储用对象存储 + 签名 URL（不能是可枚举的公开直链）
   - 提供"导出我的数据""注销账号并删除全部数据"的功能（这是后续走正规上线流程基本会被要求的）
   - 如果要公开上线，需要一份简单清晰的隐私政策说明都收集了哪些信息（账号信息、照片、位置、日记内容等）、怎么用、怎么删除——位置信息在国内属于敏感个人信息，收集前要有明确授权弹窗，不能默认开启
2. 如果计划开源：
   - 所有密钥/数据库连接信息改为环境变量，仓库里不能有任何真实密钥
   - 写清楚的 README（本地部署步骤、Docker Compose 一键启动）
   - 选一个开源协议（如 MIT），并说明这个协议对"别人拿去商用"的影响，供我决定
3. 如果计划商业化（给其他情侣付费使用）：
   - 会员/订阅体系（比如免费版相册容量有限，付费解锁更大容量、更多主题皮肤）
   - 管理后台：用户管理、异常内容举报处理流程（即使是私密空间，也要有举报机制防止被滥用，比如骚扰举报）
   - 国内上线网站/小程序涉及的备案、小程序类目审核等前置条件，请在风险清单里提一下

### 三、可选扩展模块（重点：架构上要支持"按需开关"）

不是所有用户都有同样的需求（比如不是所有情侣都养宠物），所以需要设计一套**可插拔的可选模块机制**，而不是把所有功能写死耦合在核心代码里。请在架构设计阶段专门给出这套机制的方案，建议方向（可调整）：

- **配置层**：每对情侣有一个 `enabled_modules` 配置（存成 JSON 数组或单独的关联表），决定这对情侣开启了哪些可选模块；配对成功后默认全部关闭，用户在设置页自行开启
- **数据层**：每个可选模块的数据表独立建表（带 `couple_id` 外键），不往核心表（用户、配对、相册、日记）里加字段，避免核心表越改越臃肿
- **后端层**：每个可选模块的接口收敛在独立的路由前缀下（如 `/api/modules/pet/*`），核心业务逻辑不依赖任何可选模块是否存在
- **前端层**：导航/首页根据 `enabled_modules` 动态渲染入口，用一个"模块注册表"（一份配置文件列出所有可选模块的名称、图标、入口组件）驱动渲染，而不是写死的 if/else
- **关闭后的数据处理**：用户关闭某个模块后，数据默认保留（不强制删除），只是隐藏入口，避免误关闭导致数据丢失的恐慌
- 这套机制设计好之后，**第一个落地的可选模块是"宠物栏"**，未来如果还想加别的可选内容（比如共同养的植物、健身打卡、共同的车辆保养记录等），可以照着同一套模式扩展，不需要再改动配对/相册/日记等核心逻辑

**宠物栏（首个可选模块，P1/P2 阶段实现）功能细节：**
1. 宠物档案：名字、品种、生日、头像照片，支持添加多只宠物（双方共同维护同一份档案，不分谁的宠物）
2. 宠物动态：类似人类的"状态分享"，但维度换成宠物相关——"喂食了""遛了""洗澡了""去医院了"，配文字/图片，形成宠物专属时间线，双方都能记录和查看
3. 宠物相册：可以复用相册的底层能力，单独加一个"宠物"分类标签，或者做成独立的宠物相册入口（请在计划里给出你认为更省工作量的方案）
4. 健康提醒（可选的增强项，优先级低于上面三点）：疫苗/驱虫到期提醒、体重变化简单趋势图
5. 入口：在"设置"里有"宠物栏"开关，开启后首页/导航才会出现这个模块的入口

### 四、设计与氛围要求（"高级感"的具体落地）

请在 UI/UX 部分给出明确的设计语言建议，而不是泛泛而谈，至少包含：
- 配色：避免饱和度过高的粉红堆砌，建议用低饱和度的暖色调（米白、奶咖、雾粉、暖金点缀），大面积留白
- 字体：中文选用柔和但清晰的字重搭配（避免过度圆体卡通字），强调排版的呼吸感
- 关键动效：照片上传成功、对方回应状态时给一个细腻但不夸张的微交互（比如轻微的心跳脉冲、柔光过渡），不要用廉价的爆炸表情/贴纸堆砌
- 信息层级：首页应该是"对方现在的状态/最近一条动态"作为视觉焦点，而不是功能入口宫格

### 五、技术栈建议（可调整，但请说明取舍）

- 后端：Java + Spring Boot（鉴权、配对关系、动态/日记/相册的业务逻辑），数据库用 MySQL；图片用对象存储（阿里云 OSS / 七牛云，二选一并说明理由）
- 前端：Vue3 + uniapp（如选方案①），或原生小程序框架（如选方案②）
- 实时推送：WebSocket（站内实时）+ 第三方推送通道（小程序订阅消息或 App 推送 SDK）
- 如果要做"每日话题""回忆推送"这类内容生成，可以引入轻量 LLM 调用（比如生成今日话题文案），但不是必须，可放在 P1 末尾作为可选项

### 六、非功能性要求

- 图片上传要做压缩/多分辨率处理，移动网络下体验要好
- 配对关系、日记等核心数据要有软删除/回收期，避免误删后无法恢复
- 做基础的接口鉴权和频率限制，防止恶意爆破邀请码

### 七、请输出的计划格式

1. 总体架构图（文字描述）+ 模块清单
2. 目录结构
3. 技术选型对比及结论（跨端框架、推送方案等）
4. **可选模块机制的具体设计方案**（配置层/数据层/后端/前端怎么联动，给出落地的表结构和代码组织建议）
5. P0/P1/P2 各阶段任务拆解，每项任务标注涉及模块、预估工作量、依赖关系
6. 隐私与权限设计专门成一节，说明数据隔离和加密方案
7. 风险点清单（技术风险、隐私合规风险、上线前置条件）
8. P0 阶段的详细任务拆分，精确到可以直接开始编码的颗粒度

计划确认后进入编码阶段，每完成一个任务运行测试并简要总结。

## 补充说明（不用复制给 Codex）

1. **关于"订单分享"功能**：我在提示词里明确要求做成手动登记卡片而不是自动同步外卖账号，这不是技术限制的妥协，而是更合理的产品设计——手动登记反而更有"用心"的感觉，自动同步技术上也拿不到官方授权。
2. **关于隐私**：这类产品一旦开源或对外上线，"别人能不能看到我的日记"是用户决定要不要用的第一道心理门槛，建议把隐私设计当成产品卖点来宣传，而不是事后补丁。
3. **关于"状态分享"的分寸**：已按你的要求把原来的"报备"改成完全自愿、类似微信状态的轻量分享机制——预设状态选项 + 可选配图，没有任何提醒/催促/未读未回应的压力设计，避免监控感，保留纯粹的"想分享就分享"的温馨感。
4. **关于宠物栏为什么要做成"可选模块机制"而不是直接加功能**：如果只是加一个宠物功能，最快的做法是直接在现有代码里插一段逻辑，但你提到"看看怎么扩展和维护"，说明你已经预见到以后可能还会想加别的"不是所有人都需要"的内容。直接插逻辑的方式，加得越多核心代码越乱，以后改一个核心功能（比如相册）都可能不小心牵连到宠物模块。所以我把这次需求顺势变成了一个架构决策点——让 Codex 先把"可选模块"这套机制设计出来，宠物栏只是第一个落地案例，以后想加什么都能照着同一个模式扩展，维护成本不会随功能数量线性增长。
